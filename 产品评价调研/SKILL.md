- [`../lark-shared/SKILL.md`](../lark-shared/SKILL.md) — 飞书认证、身份切换、权限处理
- [`../lark-doc/references/lark-doc-create.md`](../lark-doc/references/lark-doc-create.md) — 创建飞书 docx 的命令规范

## 任务

输入：产品名称或链接（通过 `$ARGUMENTS` 传入；为空则向用户询问）。

任务：全网检索这款产品的用户评价，按六大维度归类，写成结构化报告并写入飞书 docx。

如果只给了产品名，先用 web_search / web_fetch 找到官网或 Kickstarter 等官方源，补全产品全名、品牌、品类、价格区间，再开始评价检索。

## 信息源

全网检索这款产品的相关信息和评价，**尽可能多地**收集来自不同平台的真实用户反馈和评测者观点，每条观点要能回溯到具体平台。

**硬件产品重点抓 Kickstarter / YouTube / Reddit / Amazon 四个渠道的用户评论**——这是消费电子产品评价密度最高的来源，必须深挖。

不要浪费时间在 Trustpilot / SiteJabber（评论易被刷）、付费数据库（付费墙严重）。撞墙即止：某域名连续 web_fetch 失败就跳过，不要硬磕。

## 评价维度（必须按此六项归类，不要新增）

1. **产品好评** — 用户/评测者明确表达的优点
2. **产品差评** — 用户/评测者明确表达的缺点、Bug、缺陷
3. **未满足用户需求的地方或功能** — 用户希望产品具备但目前没有的（区别于差评：差评是"做了但做得不好"，未满足是"完全没做"）
4. **用户的付费点** — 让用户最终决定下单的核心理由
5. **用户不购买点** — 潜在用户看完后劝退的核心理由（区别于差评：差评是已购用户事后吐槽）
6. **全网各平台用户对这款产品的整体评价风向** — 按 Reddit / YouTube / Kickstarter / Amazon / 专业媒体 / 社交平台 拆分

## 强度分级

每条观点必须标 **极高 / 高 / 中 / 低** 四档强度。判断依据：**独立信息源数量 + 跨平台共识度**。3+ 平台反复提及且被定义为核心卖点/槽点 = 极高强度；仅 1 个来源提及 = 低强度。同一帖子多条回复不算多源；媒体引用 Reddit 评论不算两个独立源。

## 输出格式（钦定模板）

报告必须严格按以下骨架输出。**章节顺序、标题层级、强度档表示方式、bullet 形式都不可变**——不要发挥，不要漂移。

````markdown
# 产品基本信息

{200-400 字自然叙述简介：做什么、品牌、品类、核心硬件参数、核心卖点、价格、众筹情况；不分点}

{产品官方链接：Kickstarter / 官网 / Amazon 之一}

# 用户评论分析

## 产品好评

**极高强度**

- {短语主标题}：{完整事实陈述句，标注信息源；关键句用 <text bgcolor="light-blue">...</text> 包裹}
- {……}

**高强度**

- {……}

**中强度**

- {……}

**低强度**

- {……}

## 产品差评

（同上四档结构）

## 未满足用户需求的地方或功能

（同上四档结构）

## 用户的付费点

（同上四档结构）

## 用户不购买点

（同上四档结构）

## 全网各平台用户对这款产品的整体评价风向

<text bgcolor="light-yellow">整体风向：{一句话定性}</text>

各社区平台：

- Reddit/专业论坛：{风向 + 主要话题}
- YouTube/评测圈：{风向 + 主要话题}
- Kickstarter/众筹社区（如有）：{风向 + 主要话题}
- 亚马逊/零售平台：{风向 + 主要话题}
- 专业媒体：{具名媒体 + 评分 + 风向}
- 社交平台：{风向 + 主要话题}

总结：{2-3 句话综合判断}
````

格式硬规则：
- **markdown 内容首行直接从 `# 产品基本信息` 开始**——不要再在最前面加一行 `# 用户评论分析｜{产品名}`，那是飞书 docx 的 title 字段自动渲染，重复写会出现两份标题
- 一级章节用 `#`（仅"产品基本信息"和"用户评论分析"两个），二级章节（六大维度）用 `##`，**强度档用加粗段 `**xxx强度**`**——不要写成 `### xxx强度` 标题
- bullet 一律用 `-`，**不要用 `1.` 编号**
- 每条 bullet 都是「**短语主标题：完整事实陈述句**」的格式
- 某档强度真的没内容就跳过，不要为凑齐四档硬编
- 整体风向小节先用 `<text bgcolor="light-yellow">` 包裹一句定性，再 bullet 拆分各平台

## 参考样例（仅作写作密度参考）

下面两份样例可以用来参考写作密度、信息源引用方式、高亮使用频率。**格式细节以上方钦定模板为准**——样例之间格式有不一致的地方（如 bullet 形式、强度档表达），一律按模板写：

- 飞书 wiki：`用户评论分析｜RingConn Gen2 - AI 智能戒指`
- 飞书 wiki：`用户评论分析｜Flowtica - AI 录音笔`

## 输出：写入飞书 docx（必做）

报告 Markdown 准备好后，调 `lark-cli docs +create` 写入飞书。**这是最终交付动作，不能跳过。**

```bash
# Step 1: 把报告先写到本地临时文件 /tmp/{slug}-report.md
#         {slug} 用产品名英文/拼音简写，如 ringconn-gen2

# Step 2: 创建飞书 docx
lark-cli docs +create \
  --title "用户评论分析｜{产品名}" \
  --markdown @/tmp/{slug}-report.md \
  --wiki-space 7386596570240417795 \
  --wiki-node QIYWwaAgRibOAqkcomCcJcS5ncg \
  --as user
```

要点：
- 标题分隔符必须用**全角竖线** `｜`（不是半角 `|`）
- markdown 走 `@文件路径` 传入，避免长内容直接走命令行被 shell 转义破坏（特别是 `<text bgcolor="...">` 标签里的引号）
- 关键句用飞书 docx 高亮语法：`<text bgcolor="light-blue">…</text>`（颜色可选 light-blue / light-yellow / light-red）
- 默认目标位置是参考样例所在的 wiki 节点；用户指定别的位置则按用户指定
- 写入失败按 lark-shared 的"权限不足处理"流程修复，并把本地 `/tmp/{slug}-report.md` 路径告诉用户兜底

## 聊天里返回什么

写入成功后，**不要复述整篇报告**（已经在飞书里了）。只返回：

```
✅ 已生成飞书文档：用户评论分析｜{产品名}
📎 链接：{docx URL}

核心结论：{一句话总体风向}

主要好评（极高强度，前 2 条）：
- {……}
- {……}

主要差评（极高强度，前 2 条）：
- {……}
- {……}
```

写入失败时，把失败原因 + 本地 markdown 路径 `/tmp/{slug}-report.md` 一并告诉用户。

## 约束

- 禁止使用任何表格
- 禁止编造：没搜到的留空，不写"未能找到""信息有限"
- 禁止表态：不写"我推荐""值得买""不建议"
- 每条 bullet 是完整事实陈述句，开头加一个短语主标题（如 `无订阅费用的商业模式：`），后接完整描述并标注信息源（Reddit 用户 / YouTube 评测者 / MakeUseOf 评测 / Kickstarter 评论区等）
